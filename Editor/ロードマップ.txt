現状の重複構造（問題の整理）
現在、同じデータ（MeshContextの内容）が4つの独立したレイヤーで並列実装されている：
MeshContext (Runtime)
    │
    ├─── ModelSerializer.ToMeshDTO()     → MeshDTO (JSON/CSV, [Serializable])
    │
    ├─── RemoteProjectSerializer.WriteMeshContext() → binary (PLRS)
    │         ※ ToMeshDTO とほぼ同じフィールドを手書きでバイナリ化
    │
    ├─── MeshSummary.FromContext()       → MeshSummary (UI snapshot, readonly struct)
    │
    └─── SummaryBuilder.BuildFull()      → ModelSummary → 上とほぼ同内容
核心的な問題：

WriteMeshContext は ToMeshDTO の実質的な再実装（フィールドが一致）
ReadMeshContext は MeshContext を直接 new する（DTO経由でない）

MeshContext はライブオブジェクト（イベント・GPU索引・TypedIndices等を持つ）なので、バイナリから直接再構築するのは本来不正規


MeshSummary と remote header の役割は同一（軽量メタデータ）なのに別物


長期作戦
基本原則：DTO を唯一の中間表現にする
                    ┌────────────────────────────────────────┐
                    │         Runtime Layer                   │
                    │  ProjectContext / ModelContext           │
                    │  / MeshContext（ライブ、イベント付き）  │
                    └────────────────┬───────────────────────┘
                                     │ 唯一の変換点
                        ┌────────────┴─────────────┐
                        │     DTO Layer（統合）      │
                        │  ProjectDTO / ModelDTO     │
                        │  / MeshMetaDTO             │
                        │  / MeshGeoDTO   ← ★分割   │
                        └──────┬────────┬────────────┘
                               │        │
              ┌────────────────┘        └────────────────┐
              ▼                                          ▼
     JSON Encoder                              Binary Encoder
  (ModelSerializer)                       (RemoteSerializer)
  .mfproj / .mfmodel                       PLRH / PLRS

DTO分割：MeshDTO → MeshMetaDTO + MeshGeoDTO
現在の MeshDTO はメタデータとジオメトリが混在している。これを分割することで「ヘッダだけ先に送る」が自然に実現する：
MeshMetaDTO（軽量・メタデータのみ）
Name, Type, IsVisible, IsLocked, IsFolding, Depth, ParentIndex
MirrorType, MirrorAxis, BakedMirrorSourceIndex, HasBakedMirrorChild
BoneTransformDTO, WorldMatrix, BindPose, BoneModelRotation
IsIK, IKTargetIndex, IKLoopCount, IKLimitAngle
IsMorph, MorphName, MorphPanel, MorphParentIndex
ExcludeFromExport
MeshGeoDTO（重量・ジオメトリのみ）
MeshIndex（どのメッシュか）
Vertices[], Faces[], IsExpanded
FieldFlags（どのフィールドが入っているか）
この分割は ModelDTO（ファイル保存）にも使える。MeshDTO = MeshMetaDTO + MeshGeoDTO として後方互換を維持できる。

移行ロードマップ（無理のない段階）
Phase 0（今すぐ：分割送信の実装）

RemoteProjectSerializer の WriteMeshContext を廃止し、ModelSerializer.ToMeshDTO → binary encoding に切り替える
Remote binary = 既存 DTO の binary エンコーダとして位置づける
ReadMeshContext も MeshDTO 経由にし、ModelContext を直接 new しない

Phase 1（近い将来：DTO分割）

MeshDTO を MeshMetaDTO + MeshGeoDTO に分割
ModelSerializer（JSON）も同構造を使う → 分割送信が自動的に成立
MeshSummary を MeshMetaDTO の readonly view として統合（または廃止）

Phase 2（将来：シリアライザ統合）

ModelSerializer（JSON）・RemoteSerializer（binary）・CsvModelSerializer（CSV）の3系統を、共通の ModelEncoder インタフェースに統一
エンコーダだけ差し替える構造


分割送信プロトコルへの影響
この方針を採ると、プロトコル設計が次のように単純化される：
フレーム内容DTO対応PLRHプロジェクト情報 + 全モデルの MeshMetaDTO[]ProjectDTO + MeshMetaDTO[] の binaryPLRS1メッシュの MeshGeoDTO（FieldFlags付き）MeshGeoDTO の binary
PLRB（ボーンバンドル）は不要になる。MeshMetaDTO が軽量なので PLRH にまとめて入る。
